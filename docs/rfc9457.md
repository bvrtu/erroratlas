# RFC 9457 problem details

ErrorAtlas treats [RFC 9457 Problem Details for HTTP APIs](https://www.rfc-editor.org/rfc/rfc9457.html) as an interoperable representation of a source error contract, not as a replacement taxonomy.

## Field mapping

| RFC 9457                | ErrorAtlas                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------- |
| `type`                  | `entry.problem.type`; a stable problem-type URI when proven                           |
| `title`                 | `entry.problem.title`                                                                 |
| `status`                | `entry.status`                                                                        |
| `detail`                | `entry.problem.detail`; also used as `message` when it is the proven user-facing text |
| `instance`              | `entry.problem.instance` when static; dynamic request URIs remain unknown             |
| extension member `code` | top-level `entry.code`, the canonical ErrorAtlas identity                             |
| other extension members | `entry.problem.extensions`, limited to proven scalar values                           |

Dynamic instances, localized details, computed extensions, and conditional values are not guessed. They stay null or absent.

## OpenAPI

ErrorAtlas recognizes codes and static problem fields in `application/problem+json` schemas and examples, including local `$ref`, `const`, one-value `enum`, `default`, and `example`. It reports:

- a source problem exposed without `application/problem+json`;
- static problem fields that disagree between source/catalog and OpenAPI;
- existing missing, stale, and status drift.

## Migration

Catalog schema v1 remains readable. `erroratlas generate` writes schema v2 and adds optional `problem` objects while preserving every existing `description` and `resolution`. Checks against a v1 catalog do not fail merely because RFC 9457 fields are absent. Commit the regenerated v2 catalog when ready; no source-code migration is required.
