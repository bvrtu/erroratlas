# Current-state audit

Audited on 2026-06-21 against `main` at `e6e9f13`, the published `v0.5.0` GitHub release, and npm `latest` `0.5.0`.

## Already true in code

- ErrorAtlas is source-first and conservative: unsupported or ambiguous values remain unstructured instead of being promoted to facts.
- The scanner covers TypeScript/JavaScript, Python, Java, Dart, Swift, Go, C#, and Kotlin with AST-based syntax profiles.
- TypeScript/JavaScript has bounded local/cross-file proof, opt-in root-confined project imports, immutable destructuring, re-exports, namespace/default imports, and three-wrapper factory composition.
- Catalog schema v2 preserves human-authored descriptions and resolutions, remains readable alongside v1, and carries optional RFC 9457 problem details and occurrence evidence.
- Catalog, OpenAPI, RFC 9457 media type/field, baseline, net-new, incremental, fixer, runtime, Express, Fastify, and Next.js behavior has direct automated coverage.
- JSON scan output and catalog occurrences expose categorical `proven`/`partial` evidence chains without literal values or source text.
- CI runs Linux, macOS, and Windows across Node 20, 22, and 24. Package and composite Action jobs are separate release-surface checks.
- The published package installs from a packed tarball and exposes both the CLI and ESM entry point.
- A read-only aggregate benchmark query and a publication sanitizer already exist.

## Findings at audit start — resolved in the 0.6.0 candidate

- Tests are mostly inline source strings. There is no file-based, reviewable fixture corpus spanning all nine advertised languages and paired positive/noise/boundary behavior.
- `verify` and `prepack` do not enforce the same critical gates. `prepack` omits formatting, typecheck, coverage, and demo reproducibility.
- Coverage reports branches but does not enforce a branch threshold; the current measured branch coverage is 72.06%.
- No automated release-surface consistency check ties package version, changelog, README/Action examples, demo artifacts, and release audit together.
- Evidence is present in JSON/catalog data but omitted from human Markdown and SARIF properties. The committed demo catalog predates evidence and therefore does not demonstrate the v0.5.0 proof chain.
- `docs/architecture.md` explicitly says human Markdown omits evidence, which no longer supports the desired confidence UX.
- The committed benchmark snapshot was generated by ErrorAtlas 0.4.0, only covers repositories owned by `bvrtu`, and is not validated against a published JSON Schema in CI.
- Benchmark rows have public commit coordinates but no explicit per-repository SPDX license provenance or framework category.

## Risky or overclaim-prone statements

- “Nine supported languages” is technically true at the syntax-profile level, but should not imply equal interprocedural depth. Only TypeScript/JavaScript has bounded symbol/project resolution.
- The benchmark is real derived data, but it is not yet broad external validation and must not be presented as an industry benchmark or quality ranking.
- ErrorAtlas is production-conscious and designed for CI adoption; the repository does not yet justify “enterprise-ready.” It lacks a support/deprecation policy, completed security review, signed baseline provenance, broad external validation, and adapter compatibility matrices.
- Runtime monitoring is an embeddable local/HTTP event layer, not a hosted observability or tracing backend.

## Highest-value missing work

1. Add a file-based corpus with positive, unstructured, noise, and boundary fixtures for all supported languages, plus deeper TypeScript import/factory and RFC 9457 cases.
2. Make `verify`, `prepack`, CI, package metadata, docs examples, changelog, demo, and generated artifacts enforce one release story.
3. Carry confidence evidence into SARIF and Markdown in a compact privacy-safe form, then make the demo show it.
4. Add a versioned benchmark JSON Schema, strict privacy/schema validation, an explicit allowlist/provenance manifest, and reproducible aggregate generation.
5. Add a small external benchmark set with reviewed license provenance before making broader benchmark claims.

## Intentionally deferred or out of scope

- Hosted dashboards, alerting, storage, retention, and framework response ownership.
- FastAPI companion packaging and OpenTelemetry bridging until fixture/benchmark trust work is complete.
- Whole-program control-flow or dynamic execution.
- 1.0 stability claims before support/deprecation policy, security review, signed/provenance-aware baselines, and broader external validation.

## Recommended implementation slice

This pass should finish Phase 1 and the benchmark foundation: file-based fixtures, release consistency gates, evidence UX, demo regeneration, benchmark schema/privacy validation, allowlist/provenance tooling, a small verified external snapshot if reproducible, and truth-based launch/roadmap material. Adapter or hosted work should remain issue-sized follow-up work.

## Outcome of this pass

The recommended slice above is now implemented. The original findings remain in this document so the decision trail is reviewable: all-nine-language file fixtures and noise/boundary cases, Markdown/SARIF evidence, full prepack/release gates, benchmark v2/v3 schemas, allowlist/license provenance, a six-repository external snapshot, and truth-based draft launch material are present. The intentionally deferred items remain in the milestone roadmap.

## Candidate artifact ownership

- Authoritative inputs: `src/`, `tests/fixtures/`, `data/benchmark-allowlist.json`, `data/schemas/`, benchmark/release scripts, and human-authored docs.
- Generated but committed release surfaces: `dist/` for the composite Action and npm package, demo catalog/Markdown/SARIF, `data/external-benchmark-v3.json`, and `docs/benchmark.md`.
- Disposable local outputs: `coverage/`, `node_modules/`, temporary benchmark clones, npm caches, logs, and tarballs under `work/`. They are ignored and must not be staged.
